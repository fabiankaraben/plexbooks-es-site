Esta sección da a los recién llegados una introducción a Git intentando que lo sientan familiar en el resto del libro.

Te introducirás en los conceptos básicos que son centrales para Git, cómo Git se diferencia de otros sistemas de control de versión, y las operaciones básicas de Git tales como commit, merge y pull.

Puede ser que encuentres en esta sección cosas que no entiendas sobre Git, incluso si llevas un largo tiempo utilizándolo.

Comenzando

Puede ser desafiante comenzar a trabajar con Git en la línea de comandos si nunca antes has trabajado con comandos.

Ya que vas a estar trabajando con comandos de Git durante la mayor parte de este libro, en esta sección te daré una introducción sobre cómo hacer esto.

Existe un flujo de trabajo bastante común que sirve para explicar la mayoría de las interacciones que tendrás con Git:

  • Crear un fork de un repositorio remoto existente.
  • Copiar (o clonar) un repositorio remoto en tu computadora.
  • Crear un área de trabajo separada en el repositorio, donde puedas trabajar sin afectar a nadie.
  • Marcar los cambios para ser guardados en la copia local del repositorio.
  • Guardar los cambios en tu copia local del repositorio.
  • Sincronizar esos cambios locales con el repositorio remoto.
  • Opcionalmente, notificar al dueño del repositorio que tus cambios están listos para ser revisados.

Este capítulo te llevará a través de todos los pasos mencionados en la lista anterior para ayudarte a familiarizarte con lo básico del trabajo con Git en la línea de comandos.

A pesar de que este capítulo no explicará cada paso en detalle, te dará suficiente información para sentirte familiarizado con los repositorios de Git y flujo de trabajo básico de Git para entender mejor los capítulos siguientes.

¿Qué son los repositorios remotos?

Un repositorio remoto es simplemente una colección de todos los archivos de un proyecto, alojadas en algún otro lugar que no es tu máquina local. Estos podrían ser alojados internamente en tu red, pero lo más usual es alojarlos en servicios cloud como GitHub o GitLab.

Tener centralizado un repositorio remoto hace que sea fácil compartir y contribuir en un proyecto. En lugar de enviar los archivos a las personas interesadas, simplemente colocas esos archivos en el repositorio remoto para luego obtenerlos y ejecutarlos lo más rápido posible.

El primer paso es crear tu propia copia personal online, o fork, del repositorio remoto. Esto te da un espacio online donde trabajar y seguir las instrucciones de este libro sin afectar a ninguna de las demás personas leyendo este libro y siguiendo las mismas instrucciones.

Hacer fork a un repositorio remoto

Verás una pantalla como la siguiente:

Asegúrate de estar conectado con tu propia cuenta de usuario en GitHub, entonces haz clic en el botón Fork en la sección superior derecha de la página.

Cuando GitHub finalice de crear el fork del repositorio, verás una pantalla que se parece mucho al repositorio original, con la excepción de que ahora estás trabajando en una ubicación diferente, y eso es justo lo que necesitas.

Esta es una réplica exacta del repositorio original, con la excepción de esta copia existe en tu propia cuenta de usuario de GitHub. Esto quiere decir que puedes hacer lo que quieras con este repositorio, incluso eliminarlo, sin afectar al repositorio original.

Para comenzar a trabajar ahora necesitas clonar este repositorio a tu máquina local. Para hacer esto, necesitas la URL de este repositorio. Terminamos de trabajar con esta página por ahora, estamos listos para comenzar a trabajar con Git en la línea de comandos.

Clonar un repositorio desde GitHub

En la consola de comandos, tipea el siguiente comando sin presionar ENTER:

git clone

Luego de esto solo ingresa un espacio en blanco y luego pega la URL que tienes en el portapapeles.

Deberías ver algo muy similar a esto:

git clone git@github.com:tu-usuario-github/repo-de-practica.git

Analicemos cada parte de este comando:

  • git es el nombre de la herramienta de línea de comandos de Git. Cada interacción que tengas con Git en la línea de comandos comenzará con el comando git seguido del subcomando que necesites ejecutar.
  • clone es el nombre del subcomando que necesitas ejecutar en este paso. clone que dice a git que copie el repositorio remoto especificado a tu máquina local.
  • git@github.com:<tu-usuario-en-github>/repo-de-practica.git es la URL completa del repositorio remoto que se va a clonar. Esta incluye una referencia de acceso a GitHub, tu nombre de usuario en GitHub y el nombre del repositorio.

Ahora presiona Enter y ejecuta el comando. Git te mostrará algo de información sobre el estado y finalización del proceso:

Clonando en 'repo-de-practica'...
remote: Enumerating objects: 4, done.
remote: Counting objects: 100% (4/4), done.
remote: Compressing objects: 100% (4/4), done.
remote: Total 4 (delta 0), reused 0 (delta 0), pack-reused 0
Recibiendo objetos: 100% (4/4), listo.

Los detalles de esta salida no son importantes en este momento pero básicamente significa que la clonación se realizó con éxito y ahora tienes el repositorio en tu espacio de trabajo local.

El siguiente paso es navegar dentro del directorio del repositorio ejecutando el siguiente comando:

cd repo-de-practica

En este punto puedes abrir este directorio en tu editor de código favorito, por ejemplo Visual Studio Code. Aunque Git sirve para gestionar las versiones de cualquier tipo de contenido, no solamente código de programación.

Ahora que tienes el repositorio clonado en tu máquina local, el siguiente paso es crear un espacio de trabajo separado, o rama, o branch, donde puedas cambiar el contenido de cualquier archivo sin miedo a estropear el contenido original del repositorio.

Crear una nueva rama en Git

Una rama (branch) es conceptualmente una copia del contenido original del repositorio. De esta forma puedes trabajar en una rama sin afectar el contenido orginal del repositorio hasta que estés listo para unir los cambios con ese contenido original.

Si alguna vez copiaste un archivo antes de comenzar a editarlo para preservar el original intacto, entiendes el concepto.

En la línea de comandos ejecuta lo siguiente para crear una nueva rama:

git branch mi-nueva-rama

Analicemos las partes de este comando:

  • git es el nombre de la herramienta de línea de comandos de Git.
  • branch es el nombre del subcomando de Git para trabajar con ramas, si no se especifica más información adicional que el nombre de la rama, su comportamiento es crear una nueva rama.
  • mi-nueva-rama es el nombre de la rama que se va a crear.

El nombre de la rama suele ser descriptivo en relación a las ediciones que se van a realizar en el contenido dentro de esa rama.

Puedes ver las ramas creadas con el siguiente comando:

git branch

Esto se ve muy similar al comando anterior, pero en este caso no se especifica un nombre de rama.

Git te mostrará lo siguiente:

* main
  mi-nueva-rama

main es la copia original del repositorio, mientras que mi-nueva-rama es la rama que acabas de crear. El asterisco * en qué rama estás trabajando actualmente. Ahora mismo estás en la rama main pero esto no es lo que queremos, necesitamos cambiar a la nueva rama.

Para cambiar a la rama mi-nueva-rama ejecuta el siguiente comando:

git checkout mi-nueva-rama

Este comando se ve algo raro, tal vez esperabas ver algo como git change-branch pero sucede que Git ve esto en términos de checkout, es decir, como marcar la salida, proceso terminado y entrar a una nueva etapa.

Git te responderá con algo similar a:

Cambiado a rama 'mi-nueva-rama'

Ahora incluso puedes confirmar que sí se realizó el cambio de rema volviendo a ejecutar:

git branch

y verás esta salida:

  main
* mi-nueva-rama

Ahora el asterisco indica que la rama actual de trabajo es mi-nueva-rama.
